## 🔍 Debug Completo: Causa-Raiz de Cada Bug

### BUG 1 — Checkout retorna 404 (`/checkout/[id]` não existe) 🔴 CRÍTICO

**Sintoma:** `https://viajaedu.othebaldi.me/checkout/304f791e-...` → "página não existe"

**Investigação:**
- A rota `src/app/(b2c)/checkout/[excursion_id]/page.tsx` **existe** ✅
- O excursion `304f791e-...` existe no banco com `status = PUBLISHED` ✅
- A página faz `if (!excursion || excursion.status !== "PUBLISHED") notFound()` (linha 30)

**Hipóteses testadas:**
1. ✅ **RLS bloqueia CLIENT de ler `excursions`** → query retorna `null` → `notFound()` dispara
2. ✅ **Join com `vehicle_layouts` falha** → mesmo motivo (RLS em `vehicle_layouts`)
3. ⚠️ A tabela `excursions` tem `rls_enabled: true` confirmado no schema

**Causa-raiz:** A política RLS da tabela `excursions` provavelmente não permite `SELECT` para usuários `CLIENT` autenticados (ou para anon). A `checkout/page.tsx` usa `createClient()` com cookie de sessão, então o usuário está autenticado, MAS se a RLS exige `role = ADMIN` para leitura, o CLIENT não consegue e a query retorna null.

**Fix:** Adicionar política RLS `SELECT` em `excursions` para usuários autenticados:
```sql
CREATE POLICY "Usuários autenticados podem ver excursões publicadas"
ON excursions FOR SELECT
TO authenticated
USING (status IN ('PUBLISHED', 'SOLD_OUT', 'IN_PROGRESS'));
```
Mesma checagem para `vehicle_layouts` e `passenger_tickets` (leitura no checkout).

### BUG 2 — Após login, redireciona para `/painel` em vez do checkout 🔴 CRÍTICO

**Sintoma:** Usuário clica "Reservar" → vai para `/login?redirect=/checkout/ID` → loga → cai em `/painel`

**Investigação do fluxo em `actions.ts`:**
```
linha 39: if (!profile.cpf) → redirect("/completar-cadastro")  ← SEM o param redirect!
linha 43: if (ADMIN || AGENT) → redirect("/admin")             ← SEM o param redirect!
linha 51: const redirectUrl = formData.get("redirect")         ← lido TARDE DEMAIS
linha 54: if (redirectUrl) redirect(redirectUrl)               ← nunca chega aqui se passou pelas linhas acima
```

**Causa-raiz 1:** A leitura do `redirectUrl` acontece na linha 51, DEPOIS dos role-checks (linhas 39-48). Se o usuário sem CPF cai no `redirect("/completar-cadastro")` na linha 40, o fluxo termina ali e o `next` é perdido.

**Causa-raiz 2:** Mesmo para CLIENT com CPF, o código chega na linha 51 e redireciona corretamente — MAS se a `login` page não está passando o campo `redirect` no formData corretamente, `formData.get("redirect")` retorna null e vai para `/painel`.

**Fix:** Mover leitura do `redirectUrl` para o topo da função (antes dos role checks) e repassar como `?next=` em todos os redirects intermediários:
```typescript
// No topo da função login():
const redirectUrl = formData.get("redirect") as string | null;
const safeRedirect = redirectUrl?.startsWith("/") ? redirectUrl : null;

// Na linha 40:
redirect(`/completar-cadastro${safeRedirect ? `?next=${encodeURIComponent(safeRedirect)}` : ""}`);
```

### BUG 3 — Título da excursão ilegível (preto sobre fundo escuro) 🟠 UX

**Investigação:**
- Linha 88 de `excursao/[slug]/page.tsx`: `text-white` + `drop-shadow-[0_5px_15px_rgba(0,0,0,0.8)]`
- O gradiente é `from-black/80 via-black/40 to-black/10` (de baixo para cima)
- O texto fica no BOTTOM da div (items-end + pb-12) — deveria estar no fundo mais escuro

**Causa-raiz:** `drop-shadow` com cor preta em fundo escuro não cria contraste, apenas adiciona ruído visual. O `text-white` funciona, mas a sombra preta "some" no fundo escuro. A solução correta é usar `text-shadow` branco/claro no próprio texto, não sombra preta.

**Fix:** Substituir `drop-shadow-[...]` por um `text-stroke` via CSS inline ou adicionar uma camada de backdrop extra:
```tsx
// Trocar drop-shadow escuro por sombra com halo branco + gradiente mais forte
<h1 style={{ textShadow: "0 2px 4px rgba(0,0,0,0.9), 0 0 40px rgba(0,0,0,0.7)" }}
    className="text-4xl md:text-5xl lg:text-6xl font-extrabold text-white ...">
```
E reforçar o gradiente: `from-black/90 via-black/60 to-transparent`

### BUG 4 — Lightbox fecha só no X, não clicando fora 🟡 UX

**Investigação de `LightboxGallery.tsx`:**
- Linha 47: `<div ... onClick={closeLightbox}>` (backdrop) ✅ correto
- Linha 70: `<div ... onClick={(e) => e.stopPropagation()}>` (container da imagem) ✅ correto

**Causa-raiz:** O container interno (`w-full max-w-5xl h-full max-h-[80vh]`) ocupa quase toda a tela. Como `h-full` expande o container para a altura total do backdrop, a área clicável do backdrop fora do container é mínima ou inexistente em mobile. O `stopPropagation` no container interno impede o evento de chegar ao backdrop.

**Fix:** Trocar `stopPropagation` por verificação `e.target === e.currentTarget` no backdrop:
```tsx
// No backdrop:
onClick={(e) => { if (e.target === e.currentTarget) closeLightbox(); }}

// Remover o stopPropagation do container interno
// Adicionar body scroll lock ao abrir
```

## 🔐 Auditoria de Segurança

### SEC-01 🔴 CRITICAL — HMAC com chave pública
| Campo | Valor |
|-------|-------|
| **Arquivo** | `src/lib/auth-emails.ts:5` |
| **Código atual** | `const secret = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!` |
| **Problema** | `NEXT_PUBLIC_` = chave pública exposta no bundle do browser. Qualquer um pode gerar tokens de confirmação válidos. |
| **Fix** | Criar `EMAIL_CONFIRM_SECRET` (string aleatória de 64 chars) e usar no lugar |

### SEC-02 🟠 HIGH — Open Redirect no login
| Campo | Valor |
|-------|-------|
| **Arquivo** | `src/app/(auth)/actions.ts:54` |
| **Código atual** | `redirect(redirectUrl)` sem validação |
| **Problema** | Atacante pode forçar `redirect=https://phishing.com` via form manipulation |
| **Fix** | `if (redirectUrl && redirectUrl.startsWith("/")) redirect(redirectUrl)` |

### SEC-03 🟠 HIGH — Open Redirect no auth/callback
| Campo | Valor |
|-------|-------|
| **Arquivo** | `src/app/auth/callback/route.ts` |
| **Problema** | Mesmo padrão — `next` param não validado como path relativo |
| **Fix** | Validar `next` começa com `/` antes de usar |
